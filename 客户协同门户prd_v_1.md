# 客户协同门户系统 PRD（V1.0）

## 1. 文档信息

**项目名称：** 客户协同门户系统  
**文档类型：** 产品需求文档（PRD）  
**版本号：** V1.0  
**适用阶段：** 项目立项 / 产品设计 / 研发评审  
**目标用户：** 小型贸易公司、外贸协同场景  

---

## 2. 项目背景

### 2.1 背景说明
目前公司与客户之间的业务协同主要依赖微信、邮件、Excel、图片和零散文件传输进行管理，存在以下典型问题：

1. 客户资料、订单信息、文件资料分散，缺少统一管理平台。  
2. 产品规格、包装、标签、Logo 等确认事项缺少结构化记录，容易出现沟通遗漏。  
3. 报价、订单、合同、物流、售后等信息分散在不同沟通渠道，客户无法自助查询。  
4. 客户与内部人员之间的沟通缺少留痕，后续追溯成本高。  
5. 对于小公司而言，传统 CRM 过于复杂，角色与流程偏重，不适合快速落地。  

### 2.2 建设目标
建设一套轻量化的“客户协同门户系统”，以“管理员 + 客户”双角色为核心，聚焦订单协同、产品确认、文件共享、物流进度、消息通知等关键场景，提升客户服务效率和业务透明度。

### 2.3 产品定位
本系统不是传统重型 CRM，而是一套面向小型企业的：

**客户协同平台 / 客户订单协同系统 / 客户服务门户**

系统侧重于：
- 管理员统一维护客户、订单、文件、物流和消息
- 客户在线查看与自己相关的数据并完成确认动作
- 对关键业务节点形成可追溯留痕

## 3. 产品目标

### 3.1 业务目标
1. 建立统一的客户协同入口，提高客户体验。  
2. 将订单、产品确认、文件、物流、沟通统一到系统内管理。  
3. 降低微信、邮件往返沟通成本。  
4. 形成业务留痕，减少后续争议。  
5. 为后续扩展多角色、更多业务模块提供基础数据结构。

### 3.2 产品目标
1. 支持管理员统一管理客户与订单。  
2. 支持客户登录后仅查看自己相关数据。  
3. 支持客户在线完成产品确认及意见反馈。  
4. 支持订单相关文件上传、下载与可见性控制。  
5. 支持物流状态展示与消息提醒。  
6. 支持基础消息沟通与站内通知。

### 3.3 成功标准
1. 客户可独立完成订单查看、文件下载、确认反馈。  
2. 管理员可通过系统完成核心协同工作，无需频繁人工整理 Excel。  
3. 每个订单的确认记录、文件记录、状态变更记录可追溯。  
4. 系统能满足当前小团队使用，不依赖复杂组织架构。

## 4. 用户角色与权限

### 4.1 角色定义
本期仅设计两类角色：

#### 4.1.1 管理员
公司内部使用人员，统一承担客户维护、订单维护、文件管理、物流更新、消息回复等操作职责。

#### 4.1.2 客户
外部客户账号，仅可查看与自身有关的数据，并进行订单确认、文件查看、留言反馈等操作。

### 4.2 权限原则
1. 管理员可查看和管理全量业务数据。  
2. 客户仅可查看自身所属客户主体下的数据。  
3. 客户不可查看任何内部备注、成本、其他客户信息。  
4. 文件需具备“客户可见/不可见”控制能力。  
5. 所有关键操作需记录操作日志。

## 5. 使用场景

### 5.1 管理员核心场景
1. 创建客户信息并生成客户账号。  
2. 创建订单并维护订单商品信息。  
3. 发起产品确认任务给客户。  
4. 上传报价单、PI、合同、图片、物流单据等文件。  
5. 更新订单进度及物流状态。  
6. 接收并处理客户反馈与消息。

### 5.2 客户核心场景
1. 登录系统查看自己的订单。  
2. 查看产品规格、包装、标签等确认内容。  
3. 在线确认或提出修改意见。  
4. 下载订单相关文件。  
5. 查看发货与物流状态。  
6. 通过消息中心与管理员沟通。

## 8. 功能需求详述

## 8.1 登录与账号管理

### 8.1.1 功能说明
系统支持管理员和客户通过账号密码登录。

### 8.1.2 管理员端需求
1. 支持管理员账号登录。  
2. 支持修改密码。  
3. 支持查看登录日志。  
4. 支持退出登录。

### 8.1.3 客户端需求
1. 支持客户通过邮箱或账号登录。  
2. 支持修改密码。  
3. 支持查看基础资料。  
4. 支持账号启用/禁用状态控制。

### 8.1.4 规则要求
1. 密码需加密存储。  
2. 连续输错密码达到阈值后可临时限制登录。  
3. 不同角色登录后进入不同首页。  
4. 客户账号由管理员创建。

---

## 8.2 客户管理

### 8.2.1 功能目标
帮助管理员维护客户主体及客户账号信息。

### 8.2.2 字段建议
- 客户ID
- 客户公司名称
- 联系人姓名
- 联系邮箱
- 联系电话
- 国家/地区
- 账号名
- 账号状态
- 备注
- 创建时间

### 8.2.3 功能需求
1. 管理员可新增客户。  
2. 管理员可编辑客户资料。  
3. 管理员可启用/禁用客户账号。  
4. 管理员可查看客户名下所有订单。  
5. 管理员可通过关键词搜索客户。

### 8.2.4 业务规则
1. 每个客户账号绑定一个客户主体。  
2. 被禁用客户不可登录。  
3. 客户删除需采用逻辑删除，不物理清除历史订单。

---

## 8.3 订单管理

### 8.3.1 功能目标
统一管理客户订单及订单状态。

### 8.3.2 订单基础字段
- 订单ID
- 订单编号
- 客户ID
- 下单日期
- 币种
- 订单金额
- 预计交期
- 当前状态
- 备注
- 创建人
- 创建时间

### 8.3.3 订单商品字段
- 商品名称
- SKU/型号
- 规格描述
- 数量
- 单价
- 总价
- 产品图片
- 定制要求

### 8.3.4 功能需求
1. 管理员可创建订单。  
2. 管理员可编辑订单基础信息。  
3. 管理员可维护订单商品明细。  
4. 管理员可修改订单状态。  
5. 客户可查看自己的订单列表与订单详情。  
6. 支持按订单编号、客户名称、状态查询订单。

### 8.3.5 订单状态
建议首期采用简化状态：
- 待确认
- 已确认
- 生产中
- 已发货
- 已完成
- 异常处理中

### 8.3.6 规则要求
1. 状态变更必须留痕。  
2. 客户只能查看所属订单。  
3. 历史状态应可追溯。

---

## 8.4 产品确认管理

### 8.4.1 功能目标
将产品规格、包装、标签、Logo 等关键确认事项结构化管理，降低沟通歧义。

### 8.4.2 确认项建议
- 产品名称
- 型号/规格
- 颜色/尺寸
- 包装方式
- 标签内容
- Logo/定制要求
- 图片附件
- 备注

### 8.4.3 管理员端需求
1. 管理员可为订单创建确认任务。  
2. 管理员可录入多个确认项。  
3. 管理员可上传图片或附件作为确认依据。  
4. 管理员可查看客户确认结果。

### 8.4.4 客户端需求
1. 客户可查看待确认事项。  
2. 客户可对确认内容执行“确认”或“需要修改”。  
3. 客户可填写备注意见。  
4. 系统记录客户确认时间与确认人。

### 8.4.5 确认状态
- 待客户确认
- 客户已确认
- 客户要求修改

### 8.4.6 规则要求
1. 确认记录不可覆盖历史结果，应保留操作轨迹。  
2. 每次客户操作都需记录时间、账号、意见内容。  
3. 若客户要求修改，管理员可重新发起确认。

## 8.5 文件中心

### 8.5.1 功能目标
统一管理订单相关文件，实现集中归档与授权查看。

### 8.5.2 文件类型建议
- 报价单
- PI
- 合同
- 发票
- 装箱单
- 提单
- 产品图片
- 质检报告
- 其他附件

### 8.5.3 管理员端需求
1. 管理员可上传文件。  
2. 文件可关联到客户或订单。  
3. 可设置文件是否对客户可见。  
4. 可删除或替换文件。  
5. 可按类型、订单、客户筛选文件。

### 8.5.4 客户端需求
1. 客户可查看并下载对自己可见的文件。  
2. 客户可按订单查看相关文件。  
3. 可显示文件名称、类型、上传时间。

### 8.5.5 规则要求
1. 文件访问必须鉴权。  
2. 文件删除建议采用逻辑删除或版本替换留痕。  
3. 非客户可见文件不得在客户端暴露。

## 10. 非功能需求

### 10.1 安全性
1. 必须进行登录鉴权与会话校验。  
2. 客户数据必须按所属客户隔离。  
3. 文件下载必须进行权限校验。  
4. 密码必须加密存储。  
5. 关键操作必须记录日志。

### 10.2 可用性
1. 界面应简洁直观，适合非专业客户使用。  
2. 支持移动端自适应。  
3. 首期优先支持英文展示能力预留。

### 10.3 性能要求
1. 常用页面打开时间建议控制在 3 秒以内。  
2. 列表页支持分页查询。  
3. 文件上传下载稳定可靠。

### 10.4 可扩展性
系统需预留后续扩展能力，包括：
- 多角色扩展
- 审批流扩展
- ERP/物流系统对接
- 邮件/短信通知对接
- 多语言扩展

## 12. 业务流程说明

### 12.1 管理员发起订单协同流程
1. 管理员创建客户账号  
2. 管理员创建订单  
3. 管理员录入订单商品及确认项  
4. 管理员上传相关文件  
5. 客户登录查看并确认  
6. 管理员根据确认结果推进订单  
7. 管理员更新物流信息  
8. 双方通过消息中心持续沟通

### 12.2 客户确认流程
1. 客户登录系统  
2. 查看待确认事项  
3. 阅读产品规格及附件  
4. 执行确认或要求修改  
5. 系统记录确认结果与时间  
6. 管理员收到反馈并处理

## 14. 风险与注意事项

### 14.1 主要风险
1. 权限控制不到位，导致客户串数据。  
2. 订单状态与实际业务状态不一致。  
3. 文件分类与归档规则不统一。  
4. 页面设计过重，客户使用门槛高。  
5. 后续扩展时数据结构不合理。

### 14.2 应对建议
1. 优先设计权限与数据隔离。  
2. 订单状态先简化，避免过度设计。  
3. 文件类型与命名规则在实施前统一。  
4. 客户端首页突出“待办事项”。  
5. 首期坚持 MVP 原则，先满足核心业务闭环。

---

## 15. MVP 结论

对于当前小公司阶段，建议首期聚焦以下最小可用闭环：

1. 管理员创建客户账号  
2. 管理员创建订单并上传文件  
3. 客户登录查看订单和文件  
4. 客户在线确认产品信息  
5. 管理员更新物流进度  
6. 双方通过消息中心沟通  
7. 所有关键节点留痕

该版本即可满足公司当前主要业务协同需求，并为后续扩展打下基础。

## 16. 附录：建议术语统一

- 客户：外部合作公司或采购方  
- 管理员：公司内部统一操作人员  
- 订单：客户采购业务单据  
- 确认项：需要客户明确确认的产品或包装信息  
- 文件：与客户、订单相关的附件资料  
- 物流记录：订单发货与运输阶段信息  
- 消息：系统通知及双方沟通记录
